## Introduction

The backend is the core component of the Serverless Video-on-Demand Platform. It is responsible for receiving requests from the frontend application, generating secure upload URLs, storing video metadata, and providing the APIs required by the system.

Unlike traditional web applications, the backend in this project does not run on dedicated virtual machines. Instead, it is entirely built using managed AWS services, allowing the platform to scale automatically according to the number of incoming requests while eliminating the need for server administration.

During this section, the backend infrastructure will be implemented step by step using Amazon S3, Amazon DynamoDB, AWS Lambda, Amazon API Gateway, and AWS Identity and Access Management (IAM). After completing all configurations, the backend will be able to receive upload requests from users and prepare the information required for the automated video processing pipeline.

---

## Objectives

After completing this section, readers will be able to:

- Create Amazon S3 buckets for storing uploaded and processed videos.
- Create the DynamoDB table used to store video metadata.
- Configure IAM permissions for AWS Lambda.
- Deploy the Lambda function responsible for backend processing.
- Configure environment variables.
- Attach IAM policies required by Lambda.
- Integrate AWS Lambda with Amazon API Gateway.
- Deploy the REST API.
- Generate Presigned URLs.
- Upload videos successfully to Amazon S3.
- Verify that uploaded videos are stored correctly.

---

## Backend Deployment Workflow

The backend implementation consists of several sequential steps.

1. Create Amazon S3 Buckets.
2. Create the DynamoDB table.
3. Verify DynamoDB configuration.
4. Create the IAM Role for AWS Lambda.
5. Deploy the Lambda function.
6. Configure Lambda environment variables.
7. Attach IAM permissions.
8. Upload the Lambda source code.
9. Verify metadata creation.
10. Configure Amazon API Gateway.
11. Deploy the REST API.
12. Test API requests.
13. Generate Presigned URLs.
14. Upload videos.
15. Verify uploaded files.

Each task depends on the previous one. Therefore, the deployment should follow the order presented in this workshop.

# Create Amazon S3 Buckets

## Purpose

Amazon S3 is used as the storage layer of the platform.

Two buckets are created during this step.

The first bucket stores the original videos uploaded by users.

The second bucket stores the transcoded output generated by AWS Elemental MediaConvert.

Separating raw videos from processed media simplifies permission management and makes the processing workflow easier to maintain.

---

## Implementation

Open the **Amazon S3** service from the AWS Management Console.

Create the first bucket for raw uploaded videos.

Next, create another bucket for processed media outputs.

Both buckets should be created in the same AWS Region to simplify service integration throughout the project.

After both buckets have been created successfully, verify that they appear in the Amazon S3 console before proceeding to the next step.

## Result

After completing this step, two Amazon S3 buckets are available.

These buckets will serve as the primary storage locations throughout the remainder of the workshop.

The Raw Upload Bucket will receive videos uploaded by users, while the Processed Media Bucket will later store the HLS files generated by AWS Elemental MediaConvert.

# Create the DynamoDB Table

## Purpose

Amazon DynamoDB is used as the metadata database for the Video-on-Demand platform. Instead of storing video information inside application files, every uploaded video is recorded as an item in the **Videos** table. This allows the system to retrieve video information efficiently while supporting automatic scaling without requiring database administration.

Each video record contains information such as the video identifier, filename, upload time, processing status, output location, and other metadata required by the application.

For this project, the **Videos** table uses **videoId** as the partition key with the **String** data type.

---

## Implementation

Navigate to the **Amazon DynamoDB** service from the AWS Management Console and select **Create table**.

Configure the table using the following settings:

- Table name: **Videos**
- Partition key: **videoId**
- Data type: **String**

All remaining settings can be left as the default configuration provided by Amazon DynamoDB. AWS automatically provisions the required infrastructure, allowing the table to scale according to application traffic.

After reviewing the configuration, create the table and wait until its status changes to **Active**.

![Create DynamoDB Table](/images/5-Workshop/02-create-dynamodb-table.png)

*Figure 5.2. Creating the Videos table in Amazon DynamoDB.*

---

## Result

Once the table status becomes **Active**, the metadata database is ready for use by the backend services.

At this stage, the table does not contain any records because no videos have been uploaded yet. Metadata items will be created automatically after the backend API generates an upload request.

# Verify the DynamoDB Table

## Purpose

Before continuing with the backend deployment, the DynamoDB table should be verified to ensure that it has been created successfully and is ready to receive metadata records from AWS Lambda.

Verifying the table before deploying the backend helps identify configuration issues early and prevents failures during later stages of the implementation.

---

## Implementation

Open the **Videos** table in the DynamoDB console and review its configuration.

Verify that:

- The table status is **Active**.
- The partition key is **videoId**.
- The billing mode is configured correctly.
- No metadata records exist yet.

Because the backend has not been deployed at this stage, the table is expected to be empty.

![Verify DynamoDB Table](/images/5-Workshop/03-dynamodb-item.png)

*Figure 5.3. Verifying the newly created DynamoDB table.*

---

## Result

The DynamoDB table has been successfully created and verified.

This table will be used throughout the remainder of the workshop to store and update video metadata. Once the backend API is deployed, new items will be inserted automatically whenever users upload videos through the application.

# Create the IAM Role for AWS Lambda

## Purpose

AWS Lambda requires an execution role to interact securely with other AWS services. Instead of embedding credentials inside the application, AWS uses IAM Roles to grant temporary permissions during function execution.

In this project, the Lambda execution role is responsible for accessing Amazon S3, Amazon DynamoDB, Amazon CloudWatch, and AWS Elemental MediaConvert. Assigning permissions through IAM Roles improves security and follows the AWS best practice of least-privilege access.

---

## Implementation

Navigate to the **AWS Identity and Access Management (IAM)** service and create a new IAM Role.

Select **AWS Service** as the trusted entity and choose **Lambda** as the service that will use this role.

After creating the role, assign a meaningful name so that it can be easily identified during later configuration steps. Additional permissions will be attached after the Lambda function has been deployed.

Once the role has been created successfully, verify that it appears in the IAM Roles list before proceeding to the next step.

![Create Lambda IAM Role](/images/5-Workshop/04-create-lambda-role.png)

*Figure 5.4. Creating the execution role for AWS Lambda.*

---

## Result

The IAM Role has been successfully created and is now available for assignment during Lambda deployment.

This role will later receive the permissions required for accessing Amazon S3, Amazon DynamoDB, Amazon CloudWatch, and other AWS services used by the backend.

# Create the Lambda Function

## Purpose

AWS Lambda serves as the primary compute service of the backend. It processes incoming requests from Amazon API Gateway, generates Presigned URLs for direct uploads, records metadata in Amazon DynamoDB, and returns responses to the frontend application.

Because Lambda is a serverless compute service, AWS automatically provisions and scales execution environments according to incoming requests without requiring manual infrastructure management.

---

## Implementation

Open the **AWS Lambda** service and select **Create function**.

Choose **Author from scratch** and configure the function with the required runtime.

Assign the IAM Role created in the previous step as the execution role for this function. This allows Lambda to interact securely with the AWS resources required by the application.

After reviewing the configuration, create the function and wait until the deployment process has completed successfully.

![Create AWS Lambda Function](/images/5-Workshop/05-create-lambda.png)

*Figure 5.5. Creating the backend AWS Lambda function.*

---

## Result

The Lambda function has been deployed successfully and is ready for configuration.
